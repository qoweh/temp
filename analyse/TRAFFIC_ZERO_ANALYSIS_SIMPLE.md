# 교통량 0 원인 간단 분석

## 범위
- 확인 폴더: `analyse/20260407`, `analyse/20260413`, `analyse/data`
- 확인 기준: routeId `1050000571`, `1020003121`, `1070000091`, 비교군 `1010001001`

## 결론
- 이번 건은 **단순 호출 오류(HTTP 실패/파싱 오류)라기보다 원천 데이터의 vol 필드 특성**으로 보는 것이 타당합니다.
- 즉, 문제 노선의 `vol=0`은 "해당 시점에 실제 차량이 절대 0"이라기보다, **vol 관측값이 제공되지 않거나 0으로 들어오는 구간**일 가능성이 높습니다.

## 핵심 근거
1. 동일 API가 정상 응답
- `getRoadInfoList`, `getRoadTrafficInfoList` 모두 재확인 시 HTTP 200 / `headerCd=0`.
- 호출 실패가 원인이면 전반적으로 데이터가 깨져야 하는데, 그렇지 않음.

2. 같은 시점, 같은 API에서 노선별 패턴이 다름
- 문제 노선 3개는 `vol` 전건 0.
- 비교 노선 `1010001001`은 `vol` 0/비0 혼재.
- 즉 "API 전체 장애"가 아니라 **특정 노선의 vol 값 특성**.

3. `vol`만 0이고 다른 지표는 유효
- 문제 노선에서도 `spd`, `trvlTime`은 유효 평균값 존재.
- 교통 상태 지표 일부가 살아있는데 `vol`만 전건 0이면, 필드 단위 수집/매핑/관측 이슈 가능성이 큼.
